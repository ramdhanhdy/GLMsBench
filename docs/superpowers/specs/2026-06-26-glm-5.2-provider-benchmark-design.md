# GLM-5.2 Provider Benchmark — Design Spec

**Date:** 2026-06-26
**Status:** Approved (pending user review of this written spec)
**Approach:** A — custom harness from scratch

## 1. Purpose

Benchmark the performance of GLM-5.2 as served by two providers — the
official **Z.ai** API and a second provider **umans** (endpoint details TBD
by user) — to answer two questions:

1. Is umans as **accurate** as Z.ai, and do they **behave identically**
   (parity) for the same inputs at deterministic settings?
2. How do they compare on **latency** and **cost**, including the practical
   effect of Z.ai's frequent rate-limiting under load?

The official Z.ai API is currently in high demand and frequently rate-limits;
umans reportedly does not. This asymmetry is treated as a first-class signal
(measured and reported), not a hazard to mask.

## 2. Scope

**In scope:**
- Dimensions: latency/speed, output correctness/parity (temp=0 deterministic), cost.
- Benchmark suites (public subsets): MMLU, GSM8K, ARC-Challenge, HumanEval.
- Workload: 1 parity pass + 5 latency repeats per prompt per provider.
- Language stack: Python (async).
- Output: raw JSON per request + Markdown summary report.
- Rate-limit handling with checkpoint/resume.

**Out of scope (explicitly deferred):**
- Throughput / concurrency stress (requests/sec under load). Not requested.
- BBH (27 tasks, bespoke formats). ARC-Challenge only for reasoning.
- Interactive HTML dashboard. JSON + Markdown only.
- Judge-based / LLM-as-judge scoring. Deterministic comparison only.
- A framework-based implementation (Approaches B/C were rejected).

## 3. Architecture

Single Python package `glmsbench`, organized into single-purpose units. Each
unit has one job, a clear interface, and minimal knowledge of the others.

```
glmsbench/
├── config.py          # Load & validate config (providers, model, params, repeats, suites)
├── providers/
│   ├── client.py      # One async OpenAI-compatible streaming client; records timing + token usage
│   ├── retry.py       # Exponential backoff + jitter for 429/Retry-After and transient errors
│   └── ratelimit.py   # Per-provider semaphore + optional rpm/tpm leaky-bucket
├── datasets/
│   ├── base.py        # DatasetItem schema + Loader protocol
│   ├── mmlu.py        # 5-shot MCQ loader; gold = letter
│   ├── gsm8k.py       # 8-shot CoT loader; gold = integer
│   ├── arc.py         # ARC-Challenge 4-shot MCQ loader; gold = letter
│   └── humaneval.py   # HumanEval loader; gold = list of unit tests
├── scoring/
│   ├── exact.py       # Letter match (MMLU/ARC) with answer extraction
│   ├── numeric.py     # Numeric match (GSM8K) with extraction
│   ├── parity.py      # Compare two providers' canonical outputs + raw text
│   └── code.py        # HumanEval pass@1 via sandboxed execution
├── metrics.py         # Derive TTFT, inter-token latency, e2e, tokens/sec, cost; clean vs effective latency
├── checkpoint.py      # JSONL checkpoint read/write for resume
├── runner.py          # Orchestrates: load suite → run both providers → record results
├── reporting/
│   ├── json_out.py    # Emit per-request raw JSON
│   └── markdown.py    # Aggregate → summary tables, per-benchmark scores, parity diffs, rate-limit stats
└── sandbox/           # Isolated subprocess runner for HumanEval (timeout, no network, memory cap)
config.example.yaml    # Sample provider config
run.py                 # CLI entrypoint: --config, --suite, --out, --resume
```

### Module boundaries

- **`providers/client.py`** — knows only how to call one OpenAI-compatible
  endpoint and return `(text, raw_timing, token_usage, http_meta)`. It does
  not know which "provider" it is; the runner labels records. Provider-agnostic.
- **`datasets/*`** — each owns loading + prompt formatting + gold extraction
  + answer parsing for its benchmark. Common `DatasetItem` keeps the runner generic.
- **`scoring/*`** — pure functions: `(gold, output)` or `(output_a, output_b)`
  → structured result. No I/O, unit-testable.
- **`runner.py`** — the only place tying providers + datasets + scoring
  together. Drives the 1 parity pass + N latency repeats; emits a flat list
  of per-request records.
- **`reporting/*`** — consumes records, fully separated. Can regenerate
  reports from raw JSON without re-running the benchmark.

## 4. Config & Provider Model

Single YAML config; no hardcoded provider details (umans vs Z.ai is a config swap).

```yaml
model: glm-5.2
generation:
  temperature: 0.0
  top_p: 1.0
  max_tokens: 1024        # default; bumped per-benchmark (e.g. HumanEval 512, MCQ 5)
  seed: 42
repeats:
  parity: 1
  latency: 5
suites:
  - mmlu
  - gsm8k
  - arc
  - humaneval
suite_sizes:
  mmlu: 200
  gsm8k: 200
  arc: 200
  humaneval: 164          # whole
providers:
  zai:
    base_url: https://api.z.ai/v1
    api_key_env: ZAI_API_KEY
    model_id: glm-5.2
    pricing: { input: 1.40, output: 4.40 }   # USD per 1M tokens
    rate:
      concurrency: 2
      rpm: 60
      tpm: 500000
      retry: { max_attempts: 5, base_backoff_ms: 1000 }
  umans:
    base_url: <TBD>
    api_key_env: UMANS_API_KEY
    model_id: <TBD>
    pricing: { input: <TBD>, output: <TBD> }
    rate:
      concurrency: 6
      retry: { max_attempts: 3, base_backoff_ms: 500 }
```

### Decisions

- **API keys via env var names, never literals.** Config is safe to commit;
  `config.py` resolves `_env` → actual value at load and fails fast with a
  clear message if missing.
- **Shared `generation`** across both providers → fair parity fight (same
  temp, seed, params). Per-benchmark overrides (e.g. higher `max_tokens` for
  code) live in each dataset loader, applied by the runner.
- **Explicit `repeats`:** 1 parity capture + 5 latency samples. Parity uses
  the single temp=0 output; latency metrics averaged across the 5, reported
  with min/median/max.
- **Per-provider `pricing`** → cost is a derived metric
  (`input_tokens × pricing.input + output_tokens × pricing.output`),
  automatic once umans pricing is filled in.
- **`model_id` is per-provider** because Z.ai exposes `glm-5.2` directly while
  umans may serve it under a different id (e.g. `zai-org/GLM-5.2`).

### Validation (in `config.py`)

Exactly 2 providers; each resolvable (base_url present, env var set,
model_id set); pricing present (for the cost dimension); repeats ≥ 1;
concurrency ≥ 1; retry.max_attempts ≥ 1. Fail with one consolidated error
rather than dying mid-run.

## 5. Datasets & Prompt Formatting

Each benchmark loads as `list[DatasetItem(id, prompt, gold, meta)]` and owns
its prompt template + gold extraction + answer parser.

```python
@dataclass
class DatasetItem:
    id: str            # "mmlu-0042", "gsm8k-0133"
    prompt: str        # fully formatted, ready to send
    gold: str | list   # answer key; shape per benchmark
    meta: dict         # subject, expected_tokens_band, etc.

class Loader(Protocol):
    name: str
    def load(self, n: int) -> list[DatasetItem]: ...
```

| Benchmark | Source | Format | gold | max_tokens |
|-----------|--------|--------|------|------------|
| MMLU | `cais/mmlu`, "all" | 5-shot MCQ | letter A–D | 5 |
| GSM8K | `openai/gsm8k`, main | 8-shot CoT (`Q:...A:...#### N`) | integer | 256 |
| ARC-Challenge | `allenai/ai2_arc`, "ARC-Challenge" | 4-shot MCQ | letter A–D | 5 |
| HumanEval | HumanEval `HumanEval.jsonl` (canonical OpenAI release; load via `datasets` `openai/openai_humaneval` or vendored JSONL) | zero-shot function+docstring | list of unit tests | 512 |

### Answer extraction (shared concern)

MCQ/math models sometimes wrap answers in prose ("The answer is (C)"). Each
loader owns `parse(output) -> canonical` used by scoring — MMLU/ARC extract
the first A–D letter, GSM8K extracts the last `#### N` or last number.
Parity is compared on the **canonical** form (the meaningful comparison),
with raw-text equality recorded for completeness.

### Dependency note

Dataset loading uses `datasets` (HuggingFace) for MMLU/GSM8K/ARC and
HumanEval's JSON directly. Network fetch at load time, cached locally by HF
after first run.

## 6. Scoring

Pure functions, no I/O. Three scorers.

### 6.1 Accuracy scorer (`scoring/exact.py`, `numeric.py`)

Per-benchmark correctness, identical logic for both providers.

- **MMLU / ARC** (`exact.py`): `score_letter(gold, output)` parses first A–D
  letter, exact-matches `gold`. Returns `{correct, parsed, gold}`.
- **GSM8K** (`numeric.py`): `score_numeric(gold, output)` extracts last
  `#### N` if present else last bare number; matches `gold`. Returns
  `{correct, parsed, gold}`.

Produces per-provider accuracy rate per benchmark — the "is one quietly
worse?" signal.

### 6.2 Parity scorer (`scoring/parity.py`)

The core of this benchmark. Compares the two providers' outputs for the same
prompt at temp=0.

- **Text parity** (MMLU/GSM8K/ARC): compare **canonical answer** (parsed
  letter/number), not raw prose. Two providers serving identical weights
  should return the same answer; mismatch = real divergence. Raw-text
  equality also recorded.
- **Code parity** (HumanEval): compare structurally — does extracted code
  pass the **same** unit tests on both sides? Sub-signals: (a) pass@1 per
  provider, (b) "both pass or both fail?" agreement rate. Raw-text equality
  not relied on for code.

Per-prompt result:
```python
{
  "item_id": "mmlu-0042",
  "answer_agree": true,
  "raw_text_agree": false,
  "zai": {"correct": true, "parsed": "C"},
  "umans": {"correct": true, "parsed": "C"}
}
```

Aggregated into a **disagreement rate** per benchmark — the headline parity
number (e.g. "Z.ai and umans gave different answers on 3/200 MMLU items").

### 6.3 Code scorer (`scoring/code.py`)

HumanEval pass@1 via sandboxed execution:
- Extract first code block / function from output.
- Append canonical test cases, run in `sandbox/` (isolated subprocess, no
  network, hard timeout ~10s, memory cap).
- Result: `{passed, n_tests_passed, n_tests, error}`.
- Both providers' code run against the **same** tests → parity = same
  pass/fail outcome.

### Why split accuracy from parity

Different questions. Accuracy = "is umans as *right* as Z.ai." Parity = "do
they *behave identically*." A provider can be 100% accurate yet still diverge
on which correct letter it emits at temp=0 — a model-version/quantization
signal worth capturing separately.

## 7. Metrics, Cost & Timing Instrumentation

### Per-request capture (`providers/client.py`)

Streaming (`stream=True`) for clean TTFT. Client-side event timestamps:
```python
{
  "ttft_ms": <t_first_token - t_request_sent>,
  "inter_token_latency_ms": <median of (t_i - t_{i-1})>,
  "e2e_ms": <t_last_token - t_request_sent>,
  "output_tokens": <from stream, validated against usage>,
  "input_tokens": <from usage or local estimate>,
  "stop_reason": <from final chunk>,
  "http_status": 200,
  "error": null
}
```

Client-side timing is the only way to get honest TTFT/inter-token latency and
is directly comparable across providers (same measurement code on both).
Token counts trusted from provider `usage` when present, falling back to a
local count of streamed chunks.

### Reasoning (GLM-5.2 thinks by default)

GLM-5.2 reasons by default on both providers' OpenAI-compatible routes:
hidden `reasoning_content` streams before any visible `content`, and counts
against `max_tokens`. Confirmed empirically: at default (max) effort, a
trivial MCQ question didn't even finish reasoning within 100 tokens; the
original `max_tokens=5` for MMLU/ARC produced 100% empty outputs
(`stop_reason: "length"`, zero visible text) on both providers.

Fix: `Generation.reasoning_effort` (default `"low"`) is sent as
`reasoning_effort` in the request payload — both Z.ai and umans accept and
map it on `/chat/completions` (umans docs: "both are accepted and mapped"
between `reasoning_effort` and the Anthropic-style `thinking` object).
`Timing.reasoning_tokens` captures `usage.completion_tokens_details.reasoning_tokens`
separately from visible output, for transparency and to support comparing
reasoning levels later (see Open Items). Per-suite `max_tokens` were raised
(MMLU/ARC 5→300, GSM8K 256→768, HumanEval 512→2048) to give headroom for a
low-effort reasoning trace plus the visible answer.

### Metrics derivation (`metrics.py`)

- **Latency percentiles** per provider per benchmark: p50/p95 of TTFT,
  inter-token, e2e — across the 5 latency repeats (median with min/max
  spread, not a single noisy number).
- **Tokens/sec** = `output_tokens / (e2e_ms - ttft_ms) * 1000` (steady-state
  generation rate excluding cold-start).
- **Cost** = `(input_tokens/1M × pricing.input) + (output_tokens/1M × pricing.output)`,
  summed per provider. Direct comparison uses identical token counts where
  parity holds (typical at temp=0), so cost differences reflect **price**
  differences, not usage — the clean comparison.

### Parity pass vs latency repeats

Different requests. Parity records output for correctness/parity scoring;
latency repeats only contribute timing. Avoids 5 repeats polluting the
deterministic-output comparison, keeps token-count drift out of cost math
(cost uses the parity pass's token counts primarily, latency repeats as
sanity check).

## 8. Rate-Limit Asymmetry (first-class signal)

The Z.ai throttling vs umans headroom asymmetry is **measured and reported**,
not masked — while not corrupting the clean latency comparison.

**Concrete asymmetry (from real usage), for context:**
- **Z.ai**: less available. GLM-5.2 demand on Z.ai's infra causes more
  frequent throttling/capacity issues — the constraint shows up as
  *availability* (429s, backoff), not a hard ceiling.
- **umans**: more available, but concurrency-capped. No observed
  demand-driven outages; instead the $50/mo plan hard-caps in-flight
  requests at 4 concurrent. The constraint shows up as *throughput
  ceiling*, not intermittent failure. Empirically, 3 concurrent requests
  already sustains 100-300M tokens/day, so the 4-concurrency ceiling likely
  has untested headroom above that.

These are two different failure modes (intermittent unavailability vs. a
hard concurrency ceiling) and should be reported as such, not collapsed
into a single "rate limit" number.

### Per-provider rate config

Each provider declares its own limits (see §4 config). Each gets its own
semaphore sized to `concurrency`, plus an optional token/request leaky-bucket
if `rpm`/`tpm` are set.

### Retry/backoff (`providers/retry.py`)

Handles 429 and `Retry-After` headers with exponential backoff + jitter,
bounded by `max_attempts`. Non-429 (5xx, timeouts) get a shorter retry
budget. Retry **attempts are recorded**, not silent.

### Rate-limit metrics (first-class)

Per provider, per benchmark:
```python
{
  "requests_attempted": 205,
  "requests_succeeded": 200,
  "n_throttled_429": 14,
  "n_retried": 14,
  "total_backoff_ms": 82000,
  "avg_retries_per_request": 0.07,
  "effective_wall_clock_ms": <incl. backoff>
}
```

Reported prominently — e.g. "Z.ai hit 14 rate limits (82s total backoff);
umans hit 0."

### Two latency populations (methodological core)

- **Clean latency** (apples-to-apples inference speed): percentiles computed
  *only* from requests that succeeded on the first attempt with no backoff.
  Tells whether umans's *inference* is faster, unaffected by Z.ai's queue.
- **Effective latency** (real-world experience): wall-clock from request
  intent to final response, *including* backoff/retry. What a user under
  load feels.

Reporting both prevents two failure modes: (a) Z.ai looking artificially
slow purely due to throttling when raw inference is fine; (b) hiding the
throttle entirely. Reader sees "inference speed: comparable; real-world
latency under load: Z.ai 4× worse due to throttling."

### Resumability / checkpointing (`checkpoint.py`)

A full run is ~9000+ requests (MMLU 200 + GSM8K 200 + ARC 200 + HumanEval
164, ×2 providers, ×6 passes). The runner checkpoints completed per-request
records to JSONL after each item. A killed run resumes where it left off
instead of re-spending — especially important given Z.ai is rate-limited
and paid. `run.py` gains a `--resume` flag.

### Error handling at this layer

Timeouts, non-200, and stream errors are recorded as failed requests with
`error` set — they're *counted* (error rate per provider is itself a signal)
but excluded from clean-latency percentiles so they don't skew timing. A
request returning 0 tokens is treated as a failure.

## 9. Reporting

### JSON output (`reporting/json_out.py`)

Raw per-request records + aggregated metrics, machine-readable for later
re-analysis:
```json
{
  "run_meta": { "config": {...}, "started_at": "...", "ended_at": "..." },
  "requests": [ {provider, suite, item_id, pass_type, timings, tokens, cost, error, retries, ...} ],
  "aggregates": { per-provider per-benchmark: accuracy, parity, latency percentiles, cost, rate-limit stats }
}
```

### Markdown report (`reporting/markdown.py`)

Summary tables + per-benchmark sections:
- **Accuracy table**: per benchmark × provider → accuracy %.
- **Parity section**: disagreement rate per benchmark, with per-item diff
  samples for divergent items.
- **Latency table**: clean + effective, p50/p95 for TTFT / inter-token / e2e,
  tokens/sec — per benchmark × provider.
- **Cost table**: total + per-1M-normalized cost per provider.
- **Rate-limit section**: throttles, backoff, avg retries — the Z.ai vs
  umans asymmetry prominently.

Reports can be regenerated from raw JSON without re-running the benchmark.

## 10. CLI

```
python run.py --config config.yaml [--suite mmlu,gsm8k] [--out results/] [--resume]
```

- `--config`: required path to YAML config.
- `--suite`: optional comma-list to run a subset (default: all in config).
- `--out`: output directory for JSON + Markdown (default `results/<timestamp>/`).
- `--resume`: continue from checkpoint if present.
